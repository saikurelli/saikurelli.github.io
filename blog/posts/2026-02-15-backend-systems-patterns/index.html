Patterns for Resilient Backend Systems

✦ AI Summary

This post walks through three foundational patterns for resilient distributed services: circuit breakers to stop cascading failures, exponential backoff for safe retries, and idempotency keys to make operations safe to repeat.

Distributed systems fail. Networks partition, downstream services go down, databases hit connection limits. The question isn't whether your system will encounter failure — it's whether your system handles failure gracefully or amplifies it.

Over time I've settled on three patterns that show up repeatedly in resilient backend architectures. They're not new ideas, but they're worth internalizing because they compose well and address different failure modes.

1. Circuit Breakers

A circuit breaker sits in front of a downstream call and tracks its recent failure rate. When failures exceed a threshold it "opens" and stops forwarding requests for a cooldown period, returning a fast failure immediately instead. After the cooldown it allows a small number of test requests through — if they succeed, the circuit closes again.

The key insight is that under failure, continuing to send requests often makes things worse: you exhaust thread pools, pile up timeouts, and cascade load onto an already-struggling service. The circuit breaker sacrifices individual requests to protect the broader system.

In practice, make sure the fallback behavior is explicit. A tripped circuit should return a well-defined degraded response — an empty list, a cached value, a clear error — rather than an ambiguous timeout. This matters for your callers and for on-call debugging.

2. Exponential Backoff with Jitter

When a transient failure occurs — a rate limit, a brief network blip, a momentarily overloaded service — retrying immediately is usually wrong. Linear backoff is better. Exponential backoff is better still: double the wait time on each retry up to a cap.

But pure exponential backoff has a thundering-herd problem: if many clients failed at the same moment they'll all retry at the same future moments. Adding jitter — a random offset applied to the wait time — spreads retries out and prevents synchronized retry storms.

A simple formula: min(cap, base * 2^attempt + random(0, base)). Set base to something like 100ms, cap to a few seconds, and limit total attempts to 4–6. Beyond that, treat the operation as failed and surface the error.

3. Idempotency Keys

Retries introduce a new problem: if your retry succeeds but you don't know whether the original request succeeded, you might duplicate a mutation — charge a customer twice, create two orders, send two emails.

Idempotency keys solve this. The client generates a unique key per logical operation (a UUID is fine) and includes it in the request header or body. The server stores the key alongside the result of the first successful execution. On retry, if the server has already seen this key it returns the stored result without re-executing the operation.

The storage can be as simple as a Redis SET key result EX 86400 NX. The expiry handles key cleanup; the NX (set if not exists) flag provides atomic check-and-set semantics. For financial or inventory operations, this pattern isn't optional — it's a correctness requirement.

Putting It Together

These three patterns address different problems but work well together. The circuit breaker prevents a struggling service from being overwhelmed. Exponential backoff with jitter gives transient failures room to resolve. Idempotency keys make retry-safe operations possible.

The right implementation scope matters too. Circuit breakers belong at service-to-service boundaries. Retry logic belongs as close to the call site as possible, ideally in a shared HTTP client wrapper. Idempotency key handling belongs in the service that owns the mutating operation, not in the caller.

None of these are silver bullets, but they're a solid starting foundation for any backend service that calls other services.
